*So that* folders can filter entries

*Acceptance Criteria:*

## Performance Considerations

=== Search Performance
- Debounce 300ms to reduce re-renders
- useMemo for filtered results
- Case-insensitive comparison optimized
- Early return for empty query

=== Folder Filtering Performance
- useMemo for folder count calculations
- Efficient criteria checking (short-circuit)
- Index-based lookups for authors/labels

=== Memory Considerations
- No infinite scroll needed yet (< 10k entries)
- Virtual scrolling deferred to Phase 4+

## Risk Mitigation

=== Risk: Search Performance with Large Datasets
*Mitigation:* Debouncing, useMemo, efficient filtering logic. Virtual scrolling in future phase if needed.

=== Risk: Complex Folder Criteria UX
*Mitigation:* Clear form layout, helpful labels, section grouping, validation feedback

=== Risk: Folder + Search Interaction
*Mitigation:* Clear priority (folder first, then search), visual feedback, back button

=== Risk: Breaking Changes
*Mitigation:* Maintain backward compatibility, optional features, graceful degradation

## Lessons Learned

1. **Mock complexity:** Some tests had mock-related failures that didn't affect actual functionality. Future: simplify mocks or use integration tests.

2. **Debouncing UX:** 300ms delay works well for search without feeling laggy.

3. **Criteria complexity:** Form with multiple criteria types is complex but manageable with clear sections.

4. **Testing value:** Unit tests caught several edge cases during development.
